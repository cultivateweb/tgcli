# tgcli — User Stories и архитектура

## Контекст

`tgcli` — CLI-клиент Telegram на Go. Сейчас в репозитории собран скаффолд
(диспетчер команд на стандартной библиотеке, конфиг, заглушка `internal/telegram`).
Цель этого документа — зафиксировать **как клиент должен работать с точки зрения
пользователя** (user stories), прежде чем подключать реальный транспорт.

Зафиксированные решения (из обсуждения с пользователем):

- **Доступ:** пользовательский аккаунт через **MTProto** (библиотека `gotd/td`),
  не бот. Даёт полный доступ к личным чатам, группам, каналам и истории.
- **Режим:** гибрид — и одноразовые команды для скриптов, и интерактивный TUI.
- **Фоновый режим:** важны **обе** задачи — постоянный онлайн и быстрый вызов
  поверх всего. Значит архитектура **daemon + client**.
- **MVP:** авторизация/сессия, отправка, чтение, подписка на события.

## Архитектура (для понимания историй)

Две задачи «фона» решаются разными слоями — это ключ ко всему:

1. **Демон (резидент).** `tgcli daemon` — фоновый процесс, держит одну MTProto-сессию,
   слушает обновления, копит события, шлёт desktop-уведомления. Живёт, даже когда
   ни одно окно не открыто. Слушает локальный Unix-сокет.
2. **Клиент.** `tgcli ...` (команды) и `tgcli tui` (интерфейс) — лёгкие процессы,
   которые **подключаются (attach)** к демону по сокету и отключаются. Соединение
   с Telegram при этом не рвётся. Аналогия — `tmux` server/attach.
3. **Вызов поверх всего.** Делает не сама программа, а окружение: drop-down терминал
   (`tmux popup`, Guake/Yakuake) или горячая клавиша оконного менеджера, привязанная
   к запуску `tgcli tui`. Терминальная программа не может ловить хоткей вне фокуса —
   это всегда внешний слой.

```
                 ┌──────────────────────────────┐
   Telegram  ◄──►│  tgcli daemon (MTProto, gotd) │  держит сессию, копит события
                 │  слушает unix-socket          │
                 └──────────────┬───────────────┘
                                │ local socket (RPC/JSON)
         ┌──────────────┬───────┴────────┬────────────────┐
         ▼              ▼                ▼                 ▼
   tgcli send     tgcli read       tgcli tui         tgcli events
   (скрипт)       (скрипт)         (attach, TUI)     (стрим в stdout)
```

## Персона

**Денис, разработчик.** Живёт в терминале, хочет читать и отвечать в Telegram, не
переключаясь на телефон/десктоп-клиент; хочет дёргать клиент горячей клавишей и
получать уведомления о новых сообщениях, пока окно закрыто.

---

## Эпик 1. Авторизация и сессия

**US-1.1 — Первый вход.**
Как Денис, я хочу один раз авторизоваться по номеру телефона и коду, чтобы
дальше клиент работал без повторного входа.
*Критерии:* `tgcli auth` спрашивает телефон → код из Telegram → (при 2FA) пароль;
при успехе печатает имя аккаунта; сессия сохраняется в конфиг с правами `0600`.
Неверный код — понятная ошибка и повтор, без потери введённого телефона.

**US-1.2 — Повторный запуск без входа.**
Как Денис, я хочу, чтобы при следующих запусках клиент использовал сохранённую
сессию, чтобы не вводить код каждый раз.
*Критерии:* любая команда при валидной сессии не требует повторной авторизации;
при протухшей сессии — внятное «требуется повторный вход», а не стектрейс.

**US-1.3 — Выход.**
Как Денис, я хочу выйти из аккаунта, чтобы стереть сессию на устройстве.
*Критерии:* `tgcli auth --logout` завершает сессию на сервере, очищает `session`
в конфиге, останавливает демона, если он запущен.

**US-1.4 — Статус.**
Как Денис, я хочу проверить, под кем я залогинен и онлайн ли демон.
*Критерии:* `tgcli status` показывает аккаунт, состояние сессии, работает ли демон,
аптайм демона.

## Эпик 2. Демон (фоновый резидент)

**US-2.1 — Запуск/остановка демона.**
Как Денис, я хочу запустить фоновый процесс, держащий онлайн, чтобы не терять
события, когда окно закрыто.
*Критерии:* `tgcli daemon` поднимает резидент и unix-сокет; `tgcli daemon --stop`
аккуратно гасит; повторный запуск не плодит второй экземпляр (проверка pid/lock).

**US-2.2 — Автозапуск по требованию.**
Как Денис, я хочу, чтобы команды/TUI сами поднимали демона, если он не запущен,
чтобы не думать о порядке запуска.
*Критерии:* `tgcli send/read/tui` при отсутствии демона стартуют его в фоне и
подключаются; флаг `--no-daemon` выполняет операцию разово, без резидента.

**US-2.3 — Переподключение.**
Как Денис, я хочу, чтобы при обрыве сети демон сам восстанавливал соединение,
чтобы не делать это руками.
*Критерии:* демон переподключается с backoff; `tgcli status` отражает «оффлайн →
переподключение → онлайн».

## Эпик 3. Отправка сообщений

**US-3.1 — Отправить текст.**
Как Денис, я хочу отправить сообщение из командной строки, чтобы использовать это
в скриптах.
*Критерии:* `tgcli send --to @user "текст"`; получатель — `@username`, телефон,
chat id или `me` (Избранное); при ошибке — ненулевой код выхода и причина.

**US-3.2 — Текст из stdin / pipe.**
Как Денис, я хочу слать вывод другой команды в чат, чтобы делать алерты.
*Критерии:* `echo "сборка готова" | tgcli send --to me` — текст берётся из stdin.

**US-3.3 — Вложение (после MVP).**
Как Денис, я хочу приложить файл.
*Критерии:* `tgcli send --to @user --file ./report.pdf "вот отчёт"`.

## Эпик 4. Чтение сообщений

**US-4.1 — Список диалогов.**
Как Денис, я хочу видеть последние чаты с непрочитанными, чтобы понять, где новое.
*Критерии:* `tgcli chats` — таблица: имя, тип, число непрочитанных, время; флаг
`--unread` фильтрует только непрочитанные; `--json` для скриптов.

**US-4.2 — История чата.**
Как Денис, я хочу прочитать последние N сообщений конкретного чата.
*Критерии:* `tgcli read --chat @user --limit 20` печатает время, автора, текст;
по умолчанию помечает прочитанным, `--peek` — без пометки.

## Эпик 5. События и уведомления

**US-5.1 — Стрим новых сообщений.**
Как Денис, я хочу видеть входящие в реальном времени в терминале.
*Критерии:* `tgcli events` стримит новые сообщения в stdout (строкой и `--json`),
пока не прерван; берёт поток от демона, не открывая второе соединение.

**US-5.2 — Desktop-уведомления.**
Как Денис, я хочу системные уведомления о новых сообщениях, пока окно закрыто.
*Критерии:* демон шлёт уведомление ОС (notify-send/аналог) на новое личное
сообщение/упоминание; список «шумных» чатов настраивается, `--mute` глушит.

## Эпик 6. Интерактивный TUI + attach/detach

**US-6.1 — Полноэкранный интерфейс.**
Как Денис, я хочу TUI со списком чатов слева и перепиской справа, навигация
стрелками, ввод сообщения снизу.
*Критерии:* `tgcli tui` подключается к демону и открывает интерфейс; стрелки —
выбор чата, Enter — открыть, ввод текста + Enter — отправить; новые сообщения
прилетают в живом времени.

**US-6.2 — Detach без разрыва.**
Как Денис, я хочу закрыть окно TUI, но оставить клиент онлайн.
*Критерии:* выход из TUI (`q`/detach-хоткей) закрывает только интерфейс; демон
остаётся онлайн и продолжает ловить события и уведомлять.

**US-6.3 — Suspend в shell (MC-стиль).**
Как Денис, я хочу временно свернуть TUI в текущий shell и вернуться.
*Критерии:* по хоткею (Ctrl+Z/Ctrl+O) TUI отпускает терминал → родительский shell;
`fg` возвращает интерфейс в прежнем состоянии. (Это поведение «в фокусе», в
отличие от вызова поверх всего из US-7.)

## Эпик 7. Вызов поверх всего (drop-down / хоткей)

**US-7.1 — Мгновенный attach.**
Как Денис, я хочу, чтобы `tgcli tui` подключался к уже готовому демону мгновенно,
чтобы вызов по хоткею был без задержки на коннект.
*Критерии:* при живом демоне TUI открывается без повторного логина и без ожидания
соединения с Telegram.

**US-7.2 — Рецепты интеграции (документация, не код).**
Как Денис, я хочу готовый рецепт «выпадающего» окна по глобальной клавише.
*Критерии:* README описывает варианты — `tmux popup -E "tgcli tui"`, привязка к
горячей клавише в GNOME/KDE/sway, drop-down терминал (Guake/Yakuake) с автозапуском
`tgcli tui`. Сам глобальный хоткей настраивается в окружении, не в tgcli.

## Карта команд (целевая)

| Команда         | Назначение                                   | MVP |
|-----------------|----------------------------------------------|-----|
| `auth [--logout]` | вход/выход                                 | ✅  |
| `status`        | кто залогинен, онлайн ли демон               | ✅  |
| `daemon [--stop]` | управление фоновым резидентом              | ✅  |
| `send`          | отправить текст (`--to`, stdin)              | ✅  |
| `chats`         | список диалогов                              | ✅  |
| `read`          | история чата                                 | ✅  |
| `events`        | стрим новых сообщений                        | ✅  |
| `tui`           | интерактивный интерфейс (attach)             | ✅  |
| `version`       | версия                                       | ✅  |

После MVP: `send --file`, поиск, реакции, медиа-загрузки, контакты.

## Нефункциональные требования

- **Без второго соединения.** Все клиенты ходят через демона; одна MTProto-сессия.
- **Скриптопригодность.** У читающих команд есть `--json`; коды выхода осмысленные.
- **Безопасность.** Сессия и `api_hash` — в файле `0600`; сокет демона — `0600`.
- **Отмена.** Контекст с отменой по Ctrl+C уже заложен в `cmd/tgcli/main.go`.

## Эскиз плана реализации (для следующей итерации)

1. Подключить `gotd/td`, реализовать `internal/telegram` (login/logout/send/read/updates)
   вместо заглушки.
2. Добавить слой демона: `internal/daemon` (unix-сокет, RPC, цикл обновлений) и
   `internal/client` (подключение к сокету). Команды вызывают клиента, не транспорт напрямую.
3. Команды `status`, `daemon`, `chats`, `read`, `events` в `internal/cli`
   (регистрация в `commands()` в `internal/cli/root.go`).
4. TUI на `bubbletea` в `internal/tui`, attach к демону, suspend-хоткей.
5. README: рецепты drop-down/хоткея (US-7.2).

## Верификация

- `go vet ./... && go build ./...` — чисто.
- `tgcli auth` → реальный вход; `tgcli status` показывает онлайн.
- `echo hi | tgcli send --to me` → сообщение в Избранном.
- `tgcli daemon`, затем в другом терминале `tgcli events` ловит новое сообщение,
  приходит desktop-уведомление.
- `tgcli tui` открывается мгновенно при живом демоне; detach оставляет демона онлайн.
- `tmux popup -E "tgcli tui"` открывает/прячет окно поверх работы.
```

> Примечание: реализация — следующий шаг; сейчас зафиксированы только истории
> и архитектура для согласования.
